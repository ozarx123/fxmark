Full-stack forex/CFD broker platform — backend, frontend, infrastructure & data flows.

Node.js 24 MongoDB Atlas React 18 Socket.IO Cloud Run + Vercel Anthropic Claude

1 Overview

FXMark is a full-featured forex/CFD broker platform with real-time trading, PAMM funds, multi-level IB commissions, double-entry bookkeeping, and AI trading scaffolding.

Platform Features

  • Real-time FX trading — EURUSD, XAUUSD; live tick streaming from Twelve Data
  • Full PAMM (Percent Allocation Money Management) fund system
  • Multi-level Introducing Broker (IB) commission engine (5 levels)
  • Double-entry bookkeeping ledger (GAAP-aligned chart of accounts)
  • A-Book / B-Book risk routing (FIX protocol scaffolded)
  • AI trading strategy scaffolding (Anthropic Claude SDK integrated)
  • Role-based access: user, ib, pamm_manager, admin, superadmin
  • TP/SL auto-execution on every market tick
  • Admin panel: KYC, trading monitor, financials, IB/PAMM approvals, audit logs

6 Data Flow

6.1 Trade Execution

Client → POST /api/trading/orders └── trading-limits.service (check block / drawdown / daily loss cap) └── order.service (validate symbol, side, volume, type) └── risk-management/ai-risk-switch (A-Book vs B-Book decision) ├── A-Book: fix-engine/lp.connector → external LP via FIX protocol └── B-Book: position opened internally └── positions.service (create position record) └── margin.service (reserve margin from wallet)

6.2 Position Close

Position Close (manual or TP/SL trigger on tick) └── positions.service.closePosition() ├── Calculate P&L ├── finance/ledger.service → post trading P&L journal entry ├── wallet → credit/debit user balance ├── pamm/distribution.service → distribute P&L to all PAMM investors │ ├── Performance fee deducted first │ ├── Each investor's share posted to ledger │ └── IB commission fired for each investor's upline chain ├── ib/commission.engine → calculate & post IB commission chain └── tradeEvents → emit Socket.IO update to user room

6.3 Market Data

Twelve Data API / WebSocket └── twelveData.js (poll) OR twelveDataWebSocket.js (stream) └── cache.js (Redis + in-memory fallback) └── src/index.js tick handler ├── Broadcast tick → Socket.IO clients + /ws WebSocket ├── Frontend FxChart (RAF-batched candle update) └── TP/SL checker → auto-close qualifying positions

6.4 Authentication

POST /api/auth/register or /api/auth/login └── auth.service → bcrypt verify password └── Issue JWT access token (7d) + refresh token (30d, MongoDB) └── Refresh: POST /api/auth/refresh (JTI rotation — old token invalidated) └── Logout: DELETE /api/auth/logout (JTI deleted from DB)

7 Key Architectural Patterns

Double-Entry Bookkeeping

Every money movement posts a balanced journal (debits = credits ± 0.001 tolerance). Pre-built posting functions exist for every financial event.

Example — Trade P&L win (broker earns):
DR  Client Funds Liability (2100)   [liability reduced]
CR  Wallet Liability       (2110)   [credited to user wallet]
DR  Trading Loss Expense   (5200)   [broker cost if user wins]
CR  Trading Revenue        (4100)   [broker revenue if user loses]

Chart of accounts:
  Assets    1xxx  |  Liabilities  2xxx
  Equity    3xxx  |  Revenue      4xxx
  Expenses  5xxx

Real-Time Architecture

  • Ticks arrive from Twelve Data → emitted to all Socket.IO rooms
  • FxChart.jsx subscribes via getDatafeedSocket(), batches tick updates with requestAnimationFrame
  • Trade events emitted to user-specific rooms via tradeEvents.js

Role-Based Access Control

Roles: user → ib → pamm_manager → admin → superadmin
Frontend: ProtectedRoute checks auth + allowedRoles[]
Backend:  role middleware on admin/ib/pamm routes

Dual Database Strategy

  • MongoDB: Active primary store for all business data
  • PostgreSQL: Schema defined in database/schema.sql, configured but unused — intended for analytics/compliance

ESM Migration State

Backend uses "type": "module" (ES Modules). Some older files (ai-trading/, fix-engine/ stubs) still use CommonJS require/module.exports — migration partially complete.
